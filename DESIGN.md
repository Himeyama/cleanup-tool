# 追加仕様

## 1. リアルタイムストリーミング表示（必須）

スキャン完了を待たず、検出したファイルを逐次ブラウザへ送信する。

```
スキャン開始

↓

5件検出

↓

35件検出

↓

102件検出

↓

...
```

### 通信方式

Server-Sent Events（SSE）を採用する。

理由：

* WebSocketより実装が容易
* サーバー→ブラウザの一方向通信で十分
* HTTPだけで実装可能
* ブラウザ標準対応
* 単一Node.jsファイルでも扱いやすい

---

## 2. 仮想スクロール（Virtual Scrolling）

一覧には数十万件以上のファイルが表示される可能性がある。

DOMへ全件描画してはならない。

要件：

* 画面に表示される行のみ描画
* スクロール位置に応じて動的生成・破棄
* DOM数は100〜300行程度を維持
* ソート・フィルタ後も仮想スクロールを維持

---

## 3. Worker Threadsによるバックグラウンド処理

重い処理はWorker Threadsへ分離する。

### メインスレッド

* HTTP Server
* SSE
* API
* UI更新
* 設定管理

### Worker

* ディレクトリ探索
* ファイル情報取得
* キャッシュ解析
* 削除
* サイズ集計
* プレビュー生成

WorkerとはMessageChannelで通信する。

---

## 4. キャンセル仕様

キャンセル時

```
Worker

↓

探索停止

↓

現在処理中のみ完了

↓

即終了
```

途中まで取得した結果は保持する。

---

## 5. 一時停止

Pause時

```
探索キュー停止

↓

現在の処理終了

↓

待機
```

Resumeで継続。

---

## 6. メモリ管理

ScanResultはMapで保持。

```
Map<id, ScanResult>
```

一覧表示用に

```
FilteredResult[]
```

を生成。

フィルタ変更時に再生成する。

### AI実装時の重要な追加指示

設計書の最後に、以下のような「非機能要件」を明記しておくことをおすすめします。

* **保守性**: クラスごとに責務を厳密に分離し、高凝集・低結合を維持する。
* **拡張性**: 新しいスキャナーやプレビュー機能は既存コードを変更せず追加できる構造（Open/Closed Principle）とする。
* **パフォーマンス**: 100万ファイル規模でもUIがフリーズしないことを目標とする。
* **安全性**: 削除対象の判定はホワイトリスト方式を基本とし、危険なシステム領域は明示的に除外する。
* **品質**: JSDocを記述し、ESLintで警告が出ないコード品質を維持する。
* **実装制約**: すべての機能を単一の `app.js` に実装するが、内部ではクラス・モジュールパターンを用いて論理的に分割する。
